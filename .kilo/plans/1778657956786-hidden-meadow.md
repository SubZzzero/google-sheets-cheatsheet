# План: анализ проекта и создание рабочих файлов контекста

## Цель
Подготовить для проекта устойчивый набор assistant-facing файлов, чтобы в следующих сессиях быстро восстанавливать контекст: что это за продукт, как он устроен, где лежит логика, какие есть инварианты, риски и правила безопасных изменений.

## Ограничение текущего шага
- Сейчас активен `plan mode`.
- Разрешена запись только в этот файл.
- `AGENTS.md` и остальные рабочие файлы будут созданы на фазе реализации, не в этой фазе.

## Что уже выяснено по проекту

### 1. Назначение проекта
- Проект: `google-sheets-cheatsheet`.
- Это небольшой клиентский SPA для быстрого поиска и использования формул Google Sheets.
- Пользовательский сценарий: выбрать категорию -> открыть карточки формул -> посмотреть синтаксис/описание/пример -> скопировать синтаксис.
- Деплой-цель: Firebase Hosting.

### 2. Технологический стек
- UI: `React 19`.
- Сборка и dev-server: `Vite 8`.
- Язык: `JavaScript / JSX`.
- Стили: `CSS Modules` + глобальные `reset.css` и `style.css`.
- Линтинг: `ESLint 9`.
- Хостинг: `firebase.json`, `.firebaserc`.
- Скрипты:
  - `npm run dev`
  - `npm run build`
  - `npm run lint`
  - `npm run preview`

### 3. Структура проекта
Ключевые точки:
- `index.html` -> HTML shell.
- `src/main.jsx` -> монтирует `<App />`.
- `src/App.jsx` -> корневой UI-контроллер.
- `src/data/formulas.js` -> главный источник контента и derived-данных.
- `src/components/Home/Home.jsx` -> стартовый экран и карточки категорий.
- `src/components/Sidebar/Sidebar.jsx` -> список категорий.
- `src/components/CardList/CardList.jsx` -> список карточек выбранной категории.
- `src/components/Card/Card.jsx` -> карточка формулы и copy-to-clipboard.
- `src/components/Header/Header.jsx` -> шапка и мобильное меню.
- `src/components/Footer/Footer.jsx` -> внешняя ссылка на GitHub.

### 4. Как приложение работает
- В проекте нет роутера, backend, API и внешнего источника данных.
- Экран переключается чисто через локальное состояние `App.jsx`:
  - `activeCategory`
  - `isSidebarOpen`
- `App.jsx` импортирует `formulas`, строит список категорий, сортирует их через `CATEGORY_ORDER`, считает количество формул и решает, показывать `Home` или `CardList`.
- `CardList` рендерит карточки по текущей категории.
- `Card` отображает формулу и копирует `formula.syntax` через `navigator.clipboard`.

### 5. Данные и контракты
- Главный файл данных: `src/data/formulas.js`.
- Это не просто JSON-массив, а одновременно:
  - источник формул (`baseFormulas`)
  - словарь подсказок аргументов (`ARGUMENT_HINTS`)
  - словарь narrative-объяснений (`EXAMPLE_NARRATIVES`)
  - слой derivation (`extractSyntaxArgs`, `buildVariablesLegend`, `buildExampleDetails`)
- Экспортируемая структура `formulas` содержит `exampleDetails` с полями:
  - `purpose`
  - `variables`
  - `calculation`
  - `result`
- `Card.jsx` не ожидает отсутствия `exampleDetails`; контракт должен сохраняться.

### 6. Текущая предметная карта данных
Текущий снимок:
- Всего формул: `80`
- Категорий: `9`
- Категории:
  - `Базовые`
  - `Часто используемые`
  - `Логические`
  - `Поиск`
  - `Текст`
  - `Данные`
  - `Массивы`
  - `Дата и время`
  - `Импорт`

### 7. Важные связи между файлами
Добавление новой категории сейчас требует синхронных изменений минимум в 3 местах:
- `src/data/formulas.js`
- `src/App.jsx` (`CATEGORY_ORDER`)
- `src/components/Home/Home.jsx` (`CATEGORY_META`)

### 8. Текущие пробелы и риски
- `src/data/formulas.js` монолитный и высокорисковый для будущих правок.
- Нет тестов и runtime-валидации данных.
- Есть смешение языков интерфейса: `COPY`, `COPIED`, `Example` при основном русском UI.
- `index.html` содержит несогласованные production-метаданные:
  - `lang="en"`
  - технический `title`
  - ссылка на `/favicon.svg`, которого в видимом срезе проекта нет
- У clipboard-кнопки слабый fallback на ошибки.
- Нет URL-state и deep-linking.
- В `.kilo/context` уже есть полезная память, но нет единой входной точки `AGENTS.md`.

### 9. Что уже существует для контекста
Уже найдены assistant-facing файлы:
- `.kilo/context/project-map.md`
- `.kilo/context/data-contracts.md`
- `.kilo/context/change-log.md`
- `.kilo/context/open-questions.md`

Вывод:
- Базовая память по проекту уже начата.
- Основной пробел: нет единого индексного файла и нет компактного operational playbook для будущих изменений.

## Что нужно сделать на фазе реализации

### 1. Создать `AGENTS.md` в корне проекта
Назначение:
- единая входная точка для быстрого восстановления контекста;
- краткий обзор продукта, архитектуры, инвариантов и правил безопасной работы;
- ссылки на более глубокие файлы в `.kilo/context`.

Планируемое содержимое `AGENTS.md`:
- краткое описание продукта;
- стек и команды запуска/проверки;
- структура ключевых директорий;
- архитектура рендера и поток данных;
- high-risk файлы;
- правила для изменений данных, UI и категорий;
- список существующих context-файлов и когда какой читать.

### 2. Обновить `.kilo/context/project-map.md`
Уточнить и зафиксировать:
- актуальное дерево ключевых файлов;
- иерархию компонентов;
- ownership состояния (`App` vs `Card`);
- responsive-модель:
  - breakpoint около `900px`
  - breakpoint около `640px`
  - desktop sticky sidebar / mobile drawer
- extension points: search, routing, filters, SEO, split data files.

### 3. Обновить `.kilo/context/data-contracts.md`
Добавить/проверить:
- текущий snapshot покрытия: `80` формул / `9` категорий;
- точную структуру `baseFormulas` и экспортируемой `formulas`;
- derivation pipeline;
- coupling `name -> EXAMPLE_NARRATIVES`;
- coupling `syntax args -> ARGUMENT_HINTS`;
- факт, что у части токенов аргументов нет dedicated-hint и используется generic fallback;
- safe change rules для добавления формулы/категории.

### 4. Обновить `.kilo/context/open-questions.md`
Нормализовать накопленные вопросы по направлениям:
- локализация UI;
- SEO/meta;
- search / routing;
- валидация данных;
- декомпозиция `formulas.js`;
- централизация категорий.

### 5. Обновить `.kilo/context/change-log.md`
Добавить новую запись о текущем этапе:
- проведён повторный полный анализ проекта;
- создан `AGENTS.md` как корневая entrypoint-память;
- актуализированы context-файлы под текущее состояние репозитория.

### 6. Создать новый operational-файл `.kilo/context/development-playbook.md`
Это наиболее полезное дополнение сверх уже существующих файлов.

Назначение:
- хранить короткие, прикладные сценарии изменений без повторного анализа кода.

Планируемые разделы:
- как безопасно добавить новую формулу;
- как безопасно добавить новую категорию;
- какие файлы проверять при UI-изменениях;
- какие проверки гонять после разных типов задач;
- какие зоны проекта считать чувствительными.

Почему этот файл нужен:
- текущие `project-map.md` и `data-contracts.md` описывают систему, но не дают короткой operational-инструкции для повторяемых изменений.

## Предлагаемая структура памяти после реализации
- `AGENTS.md`
  - быстрый вход, индекс, правила, ссылки
- `.kilo/context/project-map.md`
  - архитектура, файл-карта, компонентная карта
- `.kilo/context/data-contracts.md`
  - модель данных и derivation-контракт
- `.kilo/context/development-playbook.md`
  - прикладные инструкции по типовым изменениям
- `.kilo/context/open-questions.md`
  - спорные места и продуктовые неопределённости
- `.kilo/context/change-log.md`
  - история принятых решений и обновлений контекста

## Принципы наполнения этих файлов
- Писать кратко и фактически, без длинной narrative-воды.
- Делать их assistant-facing, а не маркетинговыми.
- Не дублировать один и тот же контент в 3-4 местах.
- В `AGENTS.md` держать только высокоуровневую карту и ссылки.
- Детали контрактов держать в `data-contracts.md`.
- Повторяемые рабочие сценарии держать в `development-playbook.md`.
- Открытые решения не маскировать под факты, а складывать в `open-questions.md`.

## Порядок реализации
1. Прочитать текущие context-файлы ещё раз перед записью, чтобы обновлять их адресно.
2. Создать `AGENTS.md` как entrypoint и индекс.
3. Обновить `project-map.md` по фактической архитектуре.
4. Обновить `data-contracts.md` по текущему состоянию `formulas.js`.
5. Обновить `open-questions.md` без удаления уже полезных вопросов.
6. Создать `development-playbook.md`.
7. Обновить `change-log.md` финальной записью.
8. Проверить итоговую связность всех файлов: чтобы `AGENTS.md` ссылался на все вспомогательные документы.

## Проверка результата после реализации
Так как меняются в основном context-файлы, базовая проверка будет такой:
- убедиться, что `AGENTS.md` существует в корне;
- убедиться, что ссылки/пути в нём корректны;
- убедиться, что данные в context-файлах соответствуют реальному состоянию кода;
- не менять рабочий код приложения без отдельной задачи;
- при необходимости только прочитать `git diff`, чтобы проверить, что изменения ограничены файлами контекста.

## Что пока не входит в эту задачу
На этом этапе не планируется:
- рефакторинг `src/data/formulas.js`;
- исправление SEO/metadata;
- локализация английских UI-строк;
- внедрение search/router/tests;
- изменения визуальной части приложения.

Это будет описано как текущее состояние и возможные направления развития, но не как часть текущей реализации.

## Ожидаемый итог после выполнения плана
После фазы реализации в репозитории появится компактная, связная система project-memory:
- один главный файл входа `AGENTS.md`;
- актуализированные файлы архитектуры и контрактов;
- operational playbook для типовых изменений;
- журнал и список открытых вопросов.

Это позволит в следующих сессиях не пересобирать проект с нуля и быстрее переходить к реальным доработкам.
