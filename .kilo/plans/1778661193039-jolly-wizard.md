# Plan: Align UA/EN card results with RU and reorder categories

## Goal
1. Make the `Result` block in UA/EN cards behave like RU cards: detailed, formula-specific output instead of the current generic text.
2. Move `frequentlyUsed` to the first position on the home screen.

## Findings
- Card rendering itself is shared for all locales in `src/components/Card/Card.jsx:48-76`.
- The RU/UA/EN difference comes from data generation, not JSX.
- RU formulas already ship with rich `exampleDetails` built from `EXAMPLE_NARRATIVES` in `src/data/formulas.js:723-1090`.
- UA/EN formulas are rebuilt in `src/data/localizedFormulas.js:436-474` and currently get generic `calculation` / `result` strings from `LOCALIZED_DETAIL_TEXT` at `src/data/localizedFormulas.js:386-444`.
- Home category order is driven by `CATEGORY_ORDER` in `src/i18n.js:10-20`, then applied in `src/App.jsx:31-37`.

## Recommended Implementation

### 1. Unify localized example-details generation
- Refactor the localized detail builder so UA/EN no longer use generic fallback text for every formula.
- Preserve the existing `exampleDetails` contract:
  - `purpose`
  - `variables`
  - `calculation`
  - `result`
- Preferred approach:
  - Introduce locale-aware narrative data for `calculation` and `result` keyed by formula name.
  - Reuse the existing RU narrative structure as the source model.
  - Add EN and UA narrative dictionaries with the same coverage as RU.
  - Update `buildLocalizedDetails()` in `src/data/localizedFormulas.js` to pull per-formula narrative text first, and only fall back to generic text if a formula is missing.
- This keeps all locales visually and behaviorally aligned while avoiding mixed-language content in EN/UA.

### 2. Keep variables/purpose behavior intact
- Continue using localized `description` as `purpose` for EN/UA.
- Continue building `variables` from localized argument-hint dictionaries.
- Only replace the generic `calculation` and `result` generation path.

### 3. Reorder home categories
- Update `CATEGORY_ORDER` in `src/i18n.js` so `'frequentlyUsed'` comes before `'basic'`.
- No `Home.jsx` changes should be needed because it renders categories in the order received from `App.jsx`.

## Concrete Edit Points
- `src/data/localizedFormulas.js`
  - Add localized per-formula narrative source.
  - Update `buildLocalizedDetails()` to use narrative entries by formula name.
  - Keep generic fallback for safety.
- `src/i18n.js`
  - Reorder `CATEGORY_ORDER` to place `frequentlyUsed` first.

## Validation
1. Run `npm run lint`.
2. Run `npm run build`.
3. Manually verify in the app:
   - RU card still shows current detailed `Calculation` and `Result` text.
   - EN card shows detailed formula-specific `Calculation` and `Result`, not generic placeholders.
   - UA card shows detailed formula-specific `Calculation` and `Result`, not generic placeholders.
   - Home screen shows `Frequently Used` / `Часто вживані` / `Часто используемые` first.

## Risks
- `src/data/localizedFormulas.js` is coupled to formula names; narrative keys must stay aligned with `formula.name`.
- Missing EN/UA narrative entries would fall back to generic text unless coverage is completed.
- `Card.jsx` assumes `exampleDetails` is always complete, so the builder must continue returning all four fields.

## Notes
- A quicker shortcut would be to reuse RU narratives directly for UA/EN, but that would produce mixed-language cards and is not recommended.
- The safer product-quality change is localized narrative parity across all three locales.
