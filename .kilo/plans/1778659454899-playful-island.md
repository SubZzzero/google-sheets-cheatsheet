# Multilingual EN/UA/RU Plan

## Goal
Add app-wide multilingual support for `en`, `ua`, and `ru`, make `en` the default locale, and place a compact language switcher in the right side of the header without disrupting the current layout.

## Scope Assumptions
- Localize all visible UI text and accessibility text.
- Localize formula content shown on cards: category labels, descriptions, examples, generated variable hints, and example narrative text.
- Keep Google Sheets function names and syntax strings in their original English form.
- Avoid adding a full i18n library; use a small in-repo solution.

## Current Findings
- `src/App.jsx` uses Russian category names as both display labels and internal identifiers.
- `src/components/Home/Home.jsx` contains a large amount of inline Russian copy plus Russian-only pluralization logic.
- `src/components/Card/Card.jsx`, `CardList.jsx`, `Header.jsx`, `Sidebar.jsx`, and `Footer.jsx` all contain hardcoded UI strings or `aria-label`s.
- `src/data/formulas.js` mixes invariant formula data with Russian localized content and Russian-derived helper text.
- `index.html` is static and currently only has a generic English title; runtime locale changes do not update document language/title.

## Main Design Decision
Use a minimal custom i18n layer with two parts:
1. A small UI dictionary for component chrome.
2. A locale-aware formula builder so formula cards are generated per selected locale.

This keeps the change contained, avoids a library migration, and fixes the current coupling where translated category names would otherwise break filtering and ordering.

## Planned Architecture

### 1. Introduce stable category IDs
Replace category-as-label coupling with invariant IDs such as:
- `basic`
- `frequentlyUsed`
- `logic`
- `lookup`
- `text`
- `data`
- `arrays`
- `dateTime`
- `import`

Use these IDs for:
- formula records
- category ordering
- metadata lookups
- active category state

Render localized labels from translation maps instead of storing display text in app state.

### 2. Add a tiny locale layer
Create a minimal locale module, for example under `src/i18n/`, that provides:
- supported locales: `en`, `ua`, `ru`
- default locale: `en`
- UI string dictionary
- helper to read/persist locale from `localStorage`
- helper to expose the current locale and translated strings

Keep this lightweight: either a small React context or top-level locale state in `App` with prop passing.

### 3. Make formula derivation locale-aware
Refactor `src/data/formulas.js` so invariant data stays stable and localized text is selected by locale.

Recommended shape:
- invariant formula records keep: `id`, `name`, `syntax`, `categoryId`
- locale-specific maps provide: `categoryLabel`, `description`, `example`, argument hint text, narrative text, and fallback strings
- export `getFormulas(locale)` instead of one static `formulas` array

This lets `exampleDetails` be rebuilt per locale while preserving the current card contract.

### 4. Centralize category metadata by category ID
Move `CATEGORY_ORDER` and `CATEGORY_META` away from localized labels.

Use category IDs as keys and localize:
- category name
- category description
- any hero/grid labels tied to categories

### 5. Replace Russian-only pluralization
Move formula count formatting out of `Home.jsx` and implement locale-aware count labels using `Intl.PluralRules` or a small per-locale formatter.

### 6. Add the header language switcher
Update `Header` to accept locale props and render a compact switcher aligned to the right.

Switcher behavior:
- visible on desktop and mobile
- clear active state
- keyboard accessible
- does not interfere with the menu button
- matches current visual style

### 7. Sync document metadata with locale
When locale changes:
- update `document.documentElement.lang`
- update `document.title`

Keep `index.html` default `lang="en"` as the initial server/static value, but make runtime updates authoritative.

## File-Level Implementation Plan

### `src/App.jsx`
- Add `locale` state with default `en`.
- Hydrate locale from `localStorage` on first render.
- Persist locale changes.
- Replace `formulas` import with `getFormulas(locale)`.
- Change `activeCategory` to store stable category IDs.
- Build categories/counts from `categoryId` instead of localized labels.
- Pass locale, translation helpers, and switcher handlers into child components.
- Add effect to update document `lang` and `title`.

### `src/components/Header/Header.jsx`
- Add switcher UI on the right side.
- Localize menu button `aria-label`, logo `alt`, and product name if needed.
- Accept `locale`, `onLocaleChange`, and localized UI text via props/context.

### `src/components/Header/Header.module.css`
- Extend header layout so branding stays left and switcher sits right.
- Add compact segmented/toggle styling.
- Verify mobile spacing so menu button, logo, title, and switcher remain usable.

### `src/components/Home/Home.jsx`
- Remove inline Russian UI text into translations.
- Convert `CATEGORY_META` to ID-keyed metadata.
- Use localized category labels/descriptions from translation/formula config.
- Replace `getFormulaCountLabel()` with locale-aware formatter.
- Localize `aria-label`s and action text.

### `src/components/Sidebar/Sidebar.jsx`
- Render localized category labels from IDs.
- Localize `aria-label`s.

### `src/components/CardList/CardList.jsx`
- Localize section title, subtitle, home button, and empty state.
- Render localized active category label rather than raw state value.

### `src/components/Card/Card.jsx`
- Localize `COPY` / `COPIED`.
- Localize section labels such as syntax, description, example, variables, calculation, result.
- Keep clipboard payload equal to `formula.syntax`.

### `src/components/Footer/Footer.jsx`
- Localize `aria-label` if kept user-facing.
- Visible `GitHub` text can remain unchanged unless stylistically unified through the dictionary.

### `src/data/formulas.js`
- Refactor `baseFormulas` to use `categoryId` instead of Russian category strings.
- Separate invariant formula structure from locale text maps.
- Make `ARGUMENT_HINTS`, `EXAMPLE_NARRATIVES`, and fallback text locale-specific.
- Export `getFormulas(locale)` that returns the same runtime contract currently expected by `Card.jsx`.

### `src/main.jsx`
- Only update if a provider wrapper is introduced; otherwise likely unchanged.

### `index.html`
- Keep initial `lang="en"`.
- Optionally improve static title to a cleaner English default if desired during implementation.

## Recommended Implementation Order
1. Refactor category identity to stable `categoryId` values.
2. Introduce locale config and UI dictionary.
3. Refactor formula data export to `getFormulas(locale)` with locale-aware derived `exampleDetails`.
4. Update `App.jsx` to own locale state and wire localized formulas/categories.
5. Localize components in order: `Header`, `Home`, `Sidebar`, `CardList`, `Card`, `Footer`.
6. Add header switcher styling and verify responsive behavior.
7. Add document `lang`/title synchronization.
8. Run validation.

## Risks To Watch
- `src/data/formulas.js` is the highest-risk file because it mixes content and derivation logic.
- Any mismatch between `categoryId` references and localized label maps will break filtering/counts.
- `Card.jsx` assumes `formula.exampleDetails` is always complete; the locale-aware builder must preserve that contract.
- Mobile header space is limited; switcher styling needs to stay compact.
- Runtime document updates should not rely on `index.html` alone.

## Validation Plan
- Run `npm run lint`.
- Run `npm run build`.
- Manually verify these flows in the browser:
  - default first load uses English
  - switching locale updates header, home, sidebar, card list, and card labels
  - formula descriptions/examples/exampleDetails change with locale
  - copy button still copies syntax and feedback text localizes correctly
  - sidebar/category filtering still works after category ID refactor
  - header switcher remains usable on mobile and desktop
  - document title and `lang` attribute update with locale

## Expected Outcome
- The app defaults to English.
- Users can switch between English, Ukrainian, and Russian from a neat header control.
- UI chrome and formula-card content stay in sync with the selected locale.
- Category navigation remains stable because internal logic uses IDs, not translated labels.
